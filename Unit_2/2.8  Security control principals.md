## **Security Control Principles**

### **Definition**

Security control principles are **guidelines and best practices** that help organizations **design, implement, and manage security controls** effectively. These principles ensure **confidentiality, integrity, and availability (CIA)** of information and systems.

---

### **Objectives of Security Controls**

1. Protect **information assets** from threats.
    
2. Ensure **risk mitigation** is effective.
    
3. Provide **assurance** to stakeholders.
    
4. Support **compliance** with standards and regulations (ISO, NIST, GDPR).
    
5. Enable **continuous monitoring** and improvement.
    

---

### **Key Security Control Principles**

1. **Least Privilege**
    
    - Users/processes should have **only the access necessary** to perform their tasks.
        
    - Reduces risk if accounts are compromised.
        
    - **Example:** A clerk can view financial records but cannot approve payments.
        
2. **Defense in Depth**
    
    - Use **multiple layers of security controls** to protect systems.
        
    - If one control fails, others still provide protection.
        
    - **Example:** Firewall + Antivirus + Multi-factor authentication.
        
3. **Separation of Duties (SoD)**
    
    - Critical tasks are **divided among multiple individuals**.
        
    - Prevents fraud and errors.
        
    - **Example:** One employee initiates a payment; another approves it.
        
4. **Fail-Safe Defaults**
    
    - Default access should be **denied**; only explicitly granted access is allowed.
        
    - Ensures security if configuration fails.
        
    - **Example:** New users have no access until assigned.
        
5. **Economy of Mechanism**
    
    - Keep **security controls simple and manageable**.
        
    - Complexity increases vulnerability.
        
    - **Example:** Simple password rules vs overly complex rules causing workarounds.
        
6. **Complete Mediation**
    
    - Every **access request** should be checked against security policy.
        
    - Prevents bypassing controls.
        
    - **Example:** Each login request must be authenticated.
        
7. **Open Design**
    
    - Security should **not rely on secrecy of design**.
        
    - Systems should be secure even if design is known.
        
    - **Example:** Cryptographic algorithms are public, security depends on keys.
        
8. **Psychological Acceptability**
    
    - Controls should **not frustrate users**.
        
    - Usable controls are followed consistently.
        
    - **Example:** Easy-to-use authentication like biometrics.
        
9. **Auditability / Accountability**
    
    - Actions should be **logged and traceable** to users.
        
    - Helps in **incident investigation and compliance**.
        
    - **Example:** Logging access to sensitive files.
        
10. **Minimization of Attack Surface**
    
    - Reduce **points where attackers can exploit the system**.
        
    - Keep unnecessary services or ports **disabled**.
        
    - **Example:** Disable unused server ports.
        
11. **Resilience**
    
    - Systems should **continue functioning** during and after an attack or failure.
        
    - **Example:** Backup servers, failover mechanisms.

### **Scenario Example (Optional for 10+ Marks)**

**Scenario:** An organization stores customer data online.

- They implement **Least Privilege** so employees can only access necessary files.
    
- **Defense in Depth** is applied via firewalls, IDS, and anti-malware.
    
- **Auditability** ensures all access is logged.
    
- Result: **Data is secure, access is monitored, and risk of breach is minimized.**
